PICOBOT — REPO STRUCTURE OVERVIEW
=================================

Scopo
-----
Questo file descrive la struttura corrente del repository picobot, il ruolo
delle cartelle principali e la responsabilità architetturale delle aree più
importanti.

Top-level layout
----------------

ARCHITECTURE.md
GUIDE.md
README.md
  Documentazione principale del progetto.

Makefile
  Entry point operativo per sviluppo locale, sandbox docker, tools bootstrap,
  test e avvio del sistema.

docker/
  Materiale Docker/runtime locale.
  Contiene il Dockerfile sandbox e gli script bootstrap/entrypoint del runtime.

dev/
  Materiale operativo per sviluppo.

dev/scripts/
  Script di supporto, bootstrap, trial e manutenzione.

docs/
  Documentazione tecnica del repository e dei flow.

infra/
  Materiale infrastrutturale locale non applicativo.

infra/searxng/
  Config e supporto per il motore di ricerca locale / searxng.

picobot/
  Codice applicativo principale del progetto.

tests/
  Suite di test automatica.

tests/fixtures/
  Fixture e dati di test.

Legacy compatibility paths
--------------------------

scripts -> dev/scripts
searxng -> infra/searxng
testdata -> tests/fixtures/data

Questi symlink esistono per compatibilità temporanea con riferimenti legacy.
Nel tempo possono essere rimossi dopo aver aggiornato tutti i riferimenti.

Aree principali del package picobot/
------------------------------------

picobot/app/
  Avvio applicazione e wiring principale CLI/runtime.

picobot/agent/
  Orchestrazione ad alto livello del turno e workflow dispatcher.

picobot/bus/
  Event bus e messaggistica interna.

picobot/channels/
  Adapter delle surface (CLI, Telegram, ecc.).

picobot/config/
  Config schema, init e gestione configurazione runtime.

picobot/context/
  Layer in evoluzione per context building condiviso tra chat, KB e futuri subagent.

picobot/knowledge/
  Strato semantico/knowledge-oriented, se distinto dal retrieval puro.

picobot/memory/
  Memoria applicativa e supporto a facts/summary/session continuity.

picobot/prompts/
  Prompt helpers, language detection e prompt templates.

picobot/providers/
  Layer provider LLM / model providers.

picobot/retrieval/
  Retrieval KB/RAG locale: ingest, query, ranking, evidence retrieval.

picobot/routing/
  Routing e policy decisionali (chat, kb, podcast, yt, tool, ecc.).

picobot/runtime/
  Runtime event-driven, publisher eventi, gestione turni inbound/outbound.

picobot/sandbox/
  Supporto al backend sandbox / esecuzione isolata.

picobot/services/
  Servizi applicativi trasversali.

picobot/session/
  Persistenza sessione, state file, memory/session management.

picobot/tools/
  Tool registry e implementazioni concrete (tts, stt, web, file, python, yt, ecc.).

picobot/ui/
  Surface CLI, rendering e command layer locale.

picobot/utils/
  Utility generiche.

Directory runtime/locali
------------------------

.picobot/
  Stato locale del progetto, config runtime locale, workspace e output.
  Non è la fonte primaria di verità del codice applicativo.

outputs/
  Se presente, contiene artefatti locali/runtime.
  Idealmente gli output runtime convergono in .picobot/workspace/outputs/.

Note architetturali
-------------------

- Il codice applicativo vive in picobot/
- I dati di test vivono in tests/fixtures/
- L’infrastruttura locale vive in infra/ e docker/
- Gli script operativi vivono in dev/scripts/
- La direzione futura consigliata è consolidare un context layer condiviso
  in picobot/context/ prima di introdurre un layer subagents più ricco.

Responsabilità chiave
---------------------

routing/
  Decide dove va una richiesta.

retrieval/ + knowledge/
  Recuperano evidenze e grounding.

agent/
  Coordina il turno e i workflow.

context/
  Dovrà diventare il layer unico di costruzione del contesto.

tools/
  Boundary delle capability eseguibili.

runtime/
  Fa transitare eventi e output sulle varie surface.

Obiettivo del riordino
----------------------

Ridurre il caos top-level, separare chiaramente:
- codice applicativo
- infrastruttura locale
- script operativi
- dati di test
- documentazione
- artefatti runtime

Questo rende più semplice il prossimo passaggio: ripensare context building
e poi introdurre subagent su fondamenta più pulite.
